## 3. Построение RAG-пайплайна

**Ingest:** PDF и md читаются через `pypdf` и текстовые загрузчики → разбиение на чанки (**900** символов, overlap **120**) → запись в **ChromaDB** (`05data/rag_index/`).

**Поиск:** запрос пользователя → embedding (`sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`) → top_k фрагментов (по умолчанию 5).

**Ответ:** найденные чанки передаются в **DeepSeek API** (или LM Studio локально) с системным промптом: отвечать **только по контексту**; при отсутствии данных — фраза «Информация не найдена в базе знаний».

**Загрузка через API (`POST /upload`):** один PDF/TXT/MD добавляется в ту же базу (требование методички ДЗ-7) — это **пополнение знаний**, не конвертация файлов.

## 4. Пять ключевых результатов

1. **Собственная база PDF канала превращается в поисковый индекс** — 18 файлов модуля 1.3 → **126 чанков** в Chroma без ручной разметки.

2. **Ответы привязаны к источникам** — в `/chat` и UI под ответом видны фрагменты (имя файла, score), что снижает риск «галлюцинаций».

3. **RAG-ассистент отделён от загрузки файлов** — основной сценарий: вопрос в чате; upload — только для новых документов (отражено в UI).

4. **Off-topic контролируется промптом** — посторонний вопрос не подменяется выдуманным ответом из головы модели.

5. **Стек адаптирован под свой продукт (AIKIVAVIORA)** — вместо Gemini из TrailCamp: локальные embeddings + DeepSeek; UI `/ui/` на :8001 вместо React :5173.
